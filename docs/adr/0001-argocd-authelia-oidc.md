# ADR-0001: ArgoCD Authentication via Authelia OIDC

**Status:** Proposed  
**Date:** 2026-05-13

## Context

ArgoCD is deployed on every cluster via `terraform-helm-cluster-bootstrap`. By default it ships with local username/password authentication only. This has two problems:

1. The admin password is generated by ArgoCD itself (or set via a Helm value) and must be retrieved out-of-band after apply — inconvenient during demos and onboarding.
2. There is no group-based access control: every user is either an admin or has no access.

Authelia is already deployed as part of the cluster bootstrap when `authelia.enabled = true`. It handles nginx forward-auth for other in-cluster services (developer portal, etc.) and has a user database synced via ESO from Infisical. Authelia 0.6+ includes a native OIDC provider.

The question is how to integrate ArgoCD authentication with the existing Authelia deployment.

Options considered:

1. **nginx forward-auth** — put ArgoCD behind nginx auth-url, same as other services. ArgoCD still has its own session; Authelia gates access to the login page.
2. **ArgoCD OIDC with Authelia as provider** — configure Authelia as an OIDC identity provider, configure ArgoCD as an OIDC client. Full SSO: one Authelia session covers ArgoCD and all other services.
3. **External OIDC provider (Zitadel, Keycloak)** — deploy a dedicated OIDC provider alongside Authelia. ArgoCD and other services all use the external provider; Authelia is removed or kept only for nginx forward-auth.
4. **Static admin credentials via Infisical** — keep local auth, generate admin password in Terraform, store in Infisical. Simple, no protocol changes, but no SSO and no group-based RBAC.

## Decision

Use **option 2**: configure Authelia as a native OIDC provider and ArgoCD as an OIDC client.

The ArgoCD OIDC client is fully encapsulated within `terraform-helm-cluster-bootstrap` — no additional caller configuration is required to enable it when `authelia != null`. Additional OIDC clients (e.g. developer portal) can be registered via an optional `oidc_clients` map on the `authelia` variable.

OIDC secrets (HMAC secret, RSA signing key, per-client secrets) are generated in Terraform, stored in Infisical, and synced to Kubernetes via ESO. The ArgoCD client secret is additionally injected directly into the ArgoCD Helm release at deploy time to avoid an ESO bootstrapping race.

## Rationale

**Option 1 (nginx forward-auth)** does not give ArgoCD user identity — nginx gates access to the UI but ArgoCD still requires a local login after passing the nginx check. This is two auth prompts, not SSO.

**Option 3 (dedicated OIDC provider)** is operationally heavier: another service to run, configure, and maintain. Authelia already covers forward-auth for nginx and already has the user database and session infrastructure. Using it for OIDC avoids running two separate identity-layer services on small lab clusters.

**Option 4 (static credentials)** solves the retrieval problem but not the RBAC problem, and doesn't scale as more services need authentication.

**Option 2** uses infrastructure already present (Authelia, ESO, Infisical) and gives us genuine SSO: one login session covers ArgoCD and every other nginx-protected service. Group-based RBAC (`admins` group → `role:admin` in ArgoCD) is a natural extension of the existing users database.

Encapsulating the ArgoCD client inside the module keeps the interface clean: callers don't need to know that ArgoCD uses OIDC, only that `authelia.enabled` turns on in-cluster auth. The `oidc_clients` escape hatch on the variable handles future services without requiring module code changes.

## Implementation

**`terraform-helm-cluster-bootstrap`**

- `authelia.tf`: add `random_password` for OIDC HMAC secret and ArgoCD client secret; add `tls_private_key` (RSA 4096) for JWT signing key; add `count`-indexed `random_password` for additional clients from `var.authelia.oidc_clients`; store all in Infisical via `infisical_secret`.
- `variables.tf`: add optional `oidc_clients` map to the `authelia` object (defaults to `{}`).
- `main.tf`: when `var.authelia != null`, extend the ArgoCD Helm release with `configs.cm["oidc.config"]` (issuer, clientID, scopes), `configs.rbac` (`g, admins, role:admin`), and `configs.secret.extra` (client secret injected directly — no ESO dependency at ArgoCD startup).

**`lab-cluster.sh`**

- `bootstrap/templates/authelia.yaml`: add `identity_providers.oidc` block to the Authelia configMap; mount `authelia-oidc` k8s secret as environment variables; render ArgoCD client entry unconditionally plus any additional clients from Helm values.
- `manifests/authelia/oidc-external-secret.yaml`: new ESO ExternalSecret syncing HMAC secret, JWK private key, and per-client secrets from Infisical into `authelia-oidc`.

Secret hygiene: non-secret client metadata (redirect URIs, scopes, policy) travels through Helm values. Client secrets travel only through Infisical → ESO → mounted k8s secret → Authelia env var. They are never rendered into Terraform outputs or Helm values objects, except for the ArgoCD client secret which is injected once into `argocd-secret` by the Helm release and not surfaced further.

## Bootstrapping order

ArgoCD OIDC config is present from the first `terraform apply`. OIDC login is unavailable until Authelia is running (ArgoCD falls back to local admin login in the interim). Authelia comes up after ArgoCD syncs the cluster-bootstrap Application and ESO has synced the `authelia-oidc` secret. This means there is a window of 1–2 sync cycles after cluster creation where only local admin login works; this is acceptable.

## Consequences

- Single sign-on across ArgoCD and all nginx-protected services once Authelia is running.
- Group membership (`admins`) in the Authelia users database controls ArgoCD access level — no separate ArgoCD user management.
- New OIDC clients (e.g. developer portal) can be added by passing entries in `authelia.oidc_clients`; their secrets are provisioned automatically without changes to the module.
- All clients must be known at `terraform apply` time — dynamic client registration is not supported. This is a known structural limitation: the bootstrap module (high-level policy) depends on the concrete list of application clients (low-level details), violating the dependency inversion principle. The correct dependency direction would have each application stack register its own OIDC client independently; this is accepted as a pragmatic trade-off for the current scale.
- The HMAC secret and JWK signing key are stable across applies (generated once, stored in Infisical). Rotating them invalidates all active Authelia OIDC sessions and requires a re-apply.
- Authelia's OIDC support requires chart version ≥ 0.6.0.

## Future Considerations

The DIP violation described above points to the correct long-term architecture: replace Authelia's static OIDC provider with Zitadel. Zitadel has a Terraform provider that allows each application stack's own workspace to register its OIDC client independently (`zitadel_application_oidc`). The bootstrap module provides the Zitadel instance as an abstraction; application stacks depend on that abstraction without the bootstrap module ever needing to know about them. The dependency arrow is corrected: high-level policy (bootstrap) no longer depends on low-level details (client list).

This also resolves the secondary limitation: the Authelia users database is currently static (file-based). Zitadel provides a user management API and a developer portal integration path, enabling self-service user management without re-applying cluster infrastructure.
